# UI review plan

## Context
- Frontend is a React 19 + Vite single-page app in `frontend/`.
- Primary product UI lives in `frontend/src/App.tsx`, `frontend/src/components/*`, and `frontend/src/pages/SetupScreen.tsx`.
- Shared design-system code lives in `frontend/src/design-system/*` and Storybook is configured in `frontend/.storybook/preview.ts`.
- Review baseline: Vercel Web Interface Guidelines fetched from `https://raw.githubusercontent.com/vercel-labs/web-interface-guidelines/main/command.md`.

## Highest-priority findings to report or fix
1. Semantic actions need cleanup.
   - `frontend/src/components/Sidebar.tsx:55` uses a clickable `<div role="button">` for each conversation card while also nesting real buttons inside it. Split the selectable region into a real button/link and keep rename/delete as separate sibling buttons.
2. Several form controls are missing robust accessible naming / metadata.
   - `frontend/src/components/Composer.tsx:21` textarea relies on placeholder text only.
   - `frontend/src/components/RuntimePanel.tsx:41` and `frontend/src/components/RuntimePanel.tsx:61` generate many inputs/selects without `name`, `autocomplete`, or numeric `inputMode` defaults.
   - `frontend/src/pages/SetupScreen.tsx:270` and `frontend/src/pages/SetupScreen.tsx:307` path inputs should explicitly use `name` and `autocomplete="off"`.
3. Async UI updates are not announced.
   - `frontend/src/components/MessageList.tsx:30` streaming responses and `frontend/src/components/MessageList.tsx:64` status text need an `aria-live` region.
   - `frontend/src/pages/SetupScreen.tsx:195` install progress and `frontend/src/App.tsx:474` / `frontend/src/App.tsx:411` notices should announce updates politely.
4. Focus treatment is partially suppressed.
   - `frontend/src/styles/global.css:650` removes the composer outline without a local replacement; add a visible `:focus-visible` / `:focus-within` state to the composer shell.
5. Modal interaction polish is incomplete.
   - `frontend/src/styles/global.css:666` defines the runtime modal but does not contain overscroll as recommended for modal surfaces.
6. Copy and typography need cleanup.
   - Replace straight-dot loading strings with ellipsis in `frontend/src/App.tsx:395`, `frontend/src/components/Composer.tsx:23`, `frontend/src/components/MessageList.tsx:64`, `frontend/src/pages/SetupScreen.tsx:184`, `frontend/src/pages/SetupScreen.tsx:278`, `frontend/src/pages/SetupScreen.tsx:315`, and matching design-system text such as `frontend/src/design-system/components/ui/Skeleton.tsx:19`.
7. Design-system primitives have reusable guideline violations.
   - `frontend/src/design-system/components/ui/Button.tsx:86`, `frontend/src/design-system/components/ui/Input.tsx:100`, and `frontend/src/design-system/components/domain/Message.tsx:127` use `transition: all`.
   - `frontend/src/design-system/components/ui/Input.tsx:67` renders a visual label that is not programmatically bound to its input.
8. Document/head polish is missing.
   - `frontend/index.html` should add a matching `<meta name="theme-color">`.
   - Font loading is duplicated/misaligned between `frontend/index.html:7` and `frontend/src/styles/global.css:1`; consolidate to one source.

## Lower-priority observations
- `frontend/src/App.tsx:39`, `frontend/src/App.tsx:48`, and `frontend/src/App.tsx:49` keep conversation selection and panel/tab state entirely in React state, so the UI is not deep-linkable.
- `frontend/src/components/Sidebar.tsx:13` and `frontend/src/design-system/components/domain/Message.tsx:30` use `toLocaleString` / `toLocaleTimeString`; `Intl.DateTimeFormat` would be more explicit and consistent with the guideline.
- `frontend/index.html` / `frontend/src/App.tsx` do not provide a skip link for the main app shell.

## Execution order
1. Audit and report product-facing issues in the live app shell first.
2. Fix reusable accessibility and motion issues in shared design-system primitives.
3. Normalize copy, focus styling, modal containment, and document metadata.
4. Decide whether URL-synced state is in scope; if not, call it out as a deliberate tradeoff rather than silently changing behavior.
5. Validate with Storybook a11y, keyboard-only navigation, and `frontend` build.

## Validation checklist
- Keyboard-only traversal reaches sidebar items, composer, modal tabs, and destructive actions cleanly.
- Screen reader announcements cover streaming chat output, install progress, and error banners.
- All inputs have labels or explicit `aria-label`, plus appropriate `name` / `autocomplete` metadata.
- No remaining `transition: all` in shared UI primitives.
- Copy uses `…` consistently for loading / in-progress states.
- `npm run build` succeeds in `frontend/` after changes.
